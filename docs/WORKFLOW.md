# 协作与日志规范

## 目的

让任何新的 Codex 或开发者只依赖仓库就能回答：项目为什么存在、已经做了什么、当前结论是什么、怎样验证、接下来做什么。

## 每轮工作流程

1. 阅读 `AGENTS.md`、`project-state.json` 和最新日志。
2. 检查 Git 状态，识别并保护已有用户改动。
3. 明确本轮目标，区分已确认需求、用户示例、推测和建议。
4. 完成修改并运行与风险相称的检查。
5. 更新文档、`project-state.json` 和必要的 `CHANGELOG.md`。
6. 新增本轮日志。
7. 检查差异后再提交；只有远端和认证可用时才推送。

## 日志模板

```markdown
# YYYY-MM-DD / NNN / 主题

## 本轮目标

## 用户确认的要求

## 执行内容

## 修改文件

## 验证

## 决定与理由

## 未决问题 / 下一步

## Git 状态
```

日志描述事实，不写“应该已经成功”之类未经验证的表述。若命令失败，记录失败原因和是否影响交付。

## 需求语言

项目结论使用以下标签思考并在必要时写入日志：

- `Confirmed`：用户明确要求或确认。
- `Proposed`：为推进项目提出、尚待确认的方案。
- `Example`：用户用来说明可能性的例子，不自动进入验收。
- `Assumption`：为暂时推进而做、之后必须验证的假设。
- `Rejected`：用户已经纠正或拒绝的理解，不得再次当成需求。

## Git 约定

- 默认分支最终使用 `main`，但首次推送前先核对远端实际分支。
- 提交信息采用简短英文类型和中文说明，例如 `docs: 建立产品定义与路线图`。
- 一轮可以包含多个本地提交，但日志必须能对应到最终变更。
- 不强推、不重写用户历史、不提交大型音频和第三方二进制资产。
