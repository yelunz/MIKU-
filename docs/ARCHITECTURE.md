# 技术方案

## 总体方法

采用“中立项目模型 + 可替换分析器 + 可替换生成器 + 外部引擎适配器”的本地桌面架构。外部虚拟歌姬软件是渲染或深度编辑后端，但本项目自己的工程数据不能依赖其中任何一家。

```text
本地音频
  ↓
音频解码与快速分析
  ├→ 可选音源分离 → 音频 stem → 非破坏混音
  └→ 分 stem 转录 → 鼓点/音符/和声候选
  ↓
中立项目模型 ←→ 时间轴与多层可视化
  ↓                    ↕
歌声编排/约束求解 ← 用户编辑、锁定、撤销
  ↓
格式适配器
  ↓
MIDI / OpenUtau（验证时最新版稳定版）/ Synthesizer V Studio Pro 1.9.0 / VOCALOID6 Editor 6.13.0 / 其他目标
```

## 核心模块

### A. 音频基础层

- 音频导入、解码、重采样和缓存。
- 波形峰值和多分辨率频谱缓存，保证长音频也能流畅缩放。
- 播放、循环选区、节拍器和时间定位。
- 原始音频不可变；stem 和裁切、音量、声像、静音、独奏、时间伸缩等操作以非破坏参数保存。

### B. 分析层

每种分析器独立运行，并将结果连同算法版本、参数和置信度写入项目：

- 速度、拍点、小节线和可变速度图。
- 段落边界与能量曲线。
- 调性、和弦及和声变化。
- 候选音符或主旋律转录。
- 可选的人声/伴奏或多音轨分离。
- 在分离 stem 上运行的鼓点、贝斯、和声和多音符转录；转录事件与音频 stem 分开保存。

分析器不直接修改用户确认的数据。重新分析时生成候选差异，由用户接受、忽略或局部合并。

第一轮已建立只依赖 Python 标准库的透明基线 `tools/analyze_audio.py`，用于验证 PCM WAV 到中立 JSON 的通路。它输出测量层、推断候选、算法来源、参数、置信度、状态和警告，但不作为生产算法选型。当前夹具中速度、首拍和调性命中预期，和弦与段落精度仍未通过完整验收，因此界面只能把它们显示为可修正候选。

### C. 中立项目模型

建议的主要实体：

- `Project`：媒体引用、版本、设置和历史。
- `AudioSource`：原文件、缓存、哈希和输入类型。
- `StemAsset`：分离声部、缓存、偏移、重构/串音指标和模型来源。
- `Timeline`：时间、节拍、小节、速度和拍号映射。
- `Anchor`：sample 与 PPQ tick 双时间坐标下的共享边界，供相邻歌词、休止和音符共同引用。
- `AnalysisLayer`：段落、和弦、调性、频谱特征、候选音符及置信度。
- `NoteEvent` / `DrumEvent`：可编辑转录候选、替代值、置信度、锁定和手工覆盖。
- `LyricRegion`：选区、文本、字/音节/音素与语言。
- `RestEvent` / `BreathEvent`：明确的休止与换气，不用无法解释的轨道空缝代替。
- `VocalTrack`：主唱或和声轨。
- `VocalPhrase`：音符、对齐、呼吸和演唱参数。
- `Constraint`：用户对范围、数据或参数设置的明确限制。
- `LockState`：禁止重生成覆盖的字段和区域。
- `Revision`：一次操作的输入、输出和可回退差异。

内部工程格式先采用带版本号的 JSON 清单加缓存目录；成熟后可以封装为单一压缩包扩展名。音频默认以引用方式保存，用户选择后才复制进工程。

### D. 歌声编排层

将不同功能拆开，避免“一次生成”导致不可控：

1. 可用音域与声库约束。
2. 音符/旋律候选生成或已有旋律适配。
3. 歌词的字、音节和音素切分。
4. 音符与歌词对齐。
5. 呼吸和句间时序。
6. 和声声部。
7. 目标引擎参数映射。

每一步都读取已确认的时间轴和约束，输出可比较候选，并记录改动字段。实现可以使用规则、优化算法或模型，但对上层界面保持同一接口。

歌词处理使用可替换语言模块。中文模块负责原文、可修正读音、字/音节边界和目标音素映射；日文模块负责原文、假名读音、拍边界和目标音素映射。英文模块首版不实现。语言模块只输出中立读音数据，声库或编辑器特有音素由适配器转换。

### E. 界面层

推荐采用多轨时间轴，而不是只提供钢琴卷帘：

- 顶部：整曲导航、段落和能量总览。
- 中部：波形、频谱、节拍/小节、和弦和其他分析层，可分别开关。
- 歌词轨：按区域显示歌词，支持选区、拆分、合并和拖动。
- 歌声轨：默认用连续音高带和节奏块表达，高级模式再显示钢琴卷帘和曲线。
- 检查器：显示当前对象的精确数据、约束、置信度、锁定状态和生成历史。

P1 浏览器工作台已经把真实分析 JSON 接入上述轨道，并验证本地 WAV 播放、选区、中文/日文歌词区域、和弦纠正和项目往返。浏览器原型只验证前端数据与交互，不代表桌面壳或最终视觉已经锁定。

用户反馈确认当前仅显示波形和候选仍不够。界面下一步增加伴奏 stem 轨、律动/鼓点轨、贝斯/和声音符候选、可编辑钢琴卷帘、主唱/和声轨和参数轨；具体重构见 `docs/MULTITRACK_COMPOSITION_DESIGN.md`。

### F. 桌面桥接层

时间轴和项目状态不得直接依赖 Electron、Tauri、Node 或 Rust。前端只通过可替换 `DesktopBridge` 使用文件哈希、对象 URL/媒体引用、项目保存、原生对话框和分析进程。渲染器不得获得任意文件系统或命令执行权限。

当前 `prototype/web-workbench/desktop-bridge.js` 是浏览器实现。桌面验证阶段扩展同一接口以启动分析任务、报告进度、取消任务和保存工程。

### G. 外部适配器层

适配器只负责能力探测、字段转换、导入/导出和损失报告。优先顺序：

1. 标准 MIDI 作为最低兼容基线。
2. 为验证时最新版稳定版 OpenUtau 完成首个开放工程格式的端到端适配。首版直接生成官方公开的 USTX 0.6 文本工程，不依赖 GUI 自动化或实验性插件 API；MIDI/UST 仅作为降级交换格式。
3. 为 Synthesizer V Studio Pro 1.9.0 建立独立的 `synthv-v1.9` 适配器；基础通道使用 MIDI 与节拍/歌词辅助数据，高保真通道通过 1.9.0 实机验证后的配套脚本实现。不要直接读写未公开稳定规范的 `.svp` 内部结构。
4. VOCALOID 主适配器为 `vocaloid/6/standalone`，验收版本固定为 6.13.0 完整版。首条可靠路径使用带音符、速度、拍号和歌词事件的标准 MIDI，并显示字段损失报告；没有公开稳定格式规范时，不直接生成或修改专有 VPR/VSQX 内部结构。V5/V4/V3 只作为版本化 MIDI 降级目标。
5. 仅在明确授权和稳定性可接受时考虑界面自动化。

适配器能力按“目标编辑器版本 × 操作系统”记录。核心应用支持三个系统，不代表每个外部后端或每项适配能力在三个系统上完全相同。

Synthesizer V Studio Pro 1.9.0 的三平台适配必须区分运行形态：Windows/macOS 可以验证独立版和插件增强路径；Linux 只承诺独立应用路径，因为官方 Linux 安装程序不包含插件。其“Import as Tracks”不会导入速度数据，适配器需要显式同步速度图或在导入前给出提示。

OpenUtau 适配器需要把实际稳定版号与 USTX 格式版本写进测试记录。USTX 可以保存伴奏引用、歌词、音符、音高、颤音和参数曲线，但目标电脑是否能正确发音仍取决于已安装声库和 Phonemizer；导出时应做依赖检查，并允许创建未绑定声库的工程。

VOCALOID 适配器必须声明主版本、编辑器形态、原生格式、可读格式、MIDI 歌词能力、参数映射和已知损失。主目标 `vocaloid/6/standalone` 固定验证 6.13.0 完整版；Lite 版可做次级冒烟测试，但两轨限制不能替代完整版验收。V3/V4 的 Job Plugin 不能作为核心路线，因为 V5/V6 已取消该能力；VST/AU/ARA 也不等于第三方自动化 API。官方编辑器没有 Linux 版本，所以 Linux 只提供交换文件准备与检查，实际打开验收在 Windows/macOS 进行。

Synthesizer V 1.9.0 适配器分为 `midi-baseline`、`midi-plus-helper-script` 和 `ust-group-fallback`。配套脚本必须声明 `minEditorVersion: 0x010900` 并在运行时检查宿主版本；脚本保持 ES5 风格，只使用 1.9 已确认的基础对象。ARA 和 Voice-to-MIDI 是 1.11 才加入的能力，不得写入 1.9.0 能力矩阵。1.10+ 普通工程不能由 1.9.0 直接打开，必须由新版显式另存为 1.9.0 兼容副本。

## 技术栈决定方式

本阶段暂不锁死具体桌面框架。下一阶段用小型技术验证比较两条路线：

- Web 技术桌面壳 + 本地分析服务：界面迭代快，适合复杂时间轴；需要处理打包和进程通信。
- 原生跨平台界面 + 本地分析库：部署路径直接；复杂交互的开发效率需要验证。

选择依据依次为：时间轴性能、本地音频能力、Windows/macOS/Linux 打包、外部进程/文件交互、可测试性和团队维护成本，而不是框架热度。

截至 2026-07-20 的官方资料核对后，Electron 43.x 是第一桌面验证候选，Tauri 2.11.x 是体积优化备选；这不是最终锁栈。统一 Chromium 和现有 Python CLI 使 Electron 的第一轮接入更直接，但必须同时验证安装包体积、内存、安全升级和沙箱。完整依据与三平台矩阵见 `docs/DESKTOP_STACK_SPIKE.md`。

技术验证和持续集成必须建立三个系统的构建矩阵。平台专用适配代码放在适配器边界内，不得渗透到时间轴、项目模型和歌声编排核心。

## 关键质量要求

- 本地优先与隐私：默认不上传用户音频。
- 可解释：分析和生成结果保存来源、参数和置信度。
- 非破坏：用户编辑不会被静默覆盖。
- 可恢复：项目自动保存，操作可撤销，缓存可重建。
- 可扩展：分析器、生成器和歌声引擎均可替换。
- 性能：长音频异步分析，界面可先展示已完成层，不等待全部任务。
