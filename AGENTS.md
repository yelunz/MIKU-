# AGENTS.md

本文件是所有 Codex 和开发者接手项目时的入口说明。

## 接手顺序

开始工作前按顺序阅读：

1. `README.md`
2. `project-state.json`
3. `docs/PRODUCT_DEFINITION.md`
4. `docs/ARCHITECTURE.md`
5. `docs/ROADMAP.md`
6. `docs/WORKFLOW.md`
7. `logs/` 中编号最大的最新日志

若代码、文档和聊天记录互相冲突，以用户最新明确确认的要求为最高优先级，并在新日志中记录冲突与处理结果。

## 当前产品边界

- 软件提供通用的音频分析、可视化、歌词分配、歌声编排、参数编辑和导出能力。
- 软件不替用户规定审美、风格、唱法或创作目的。
- 用户举出的某个使用例子不是需求，除非用户明确确认要将它纳入产品能力或验收条件。
- 不得把“面向小白”理解为隐藏或删除专业数据。应采用渐进式呈现：默认界面直观，详细数据仍可查看和修改。
- 不得假设导入的伴奏中一定存在可提取的人声旋律。分析、转录与生成是不同能力。

## 每轮工作必须留下记录

每次 Codex 对项目产生有效工作结果，包括新增产品结论、修改文档、代码、测试、资源或配置，都必须：

1. 在 `logs/` 新增一份按日期和序号命名的日志。
2. 日志写明目标、执行内容、修改文件、验证结果、决定、未决问题和 Git 状态。
3. 更新 `project-state.json` 的 `last_updated`、当前阶段、已完成项和下一步。
4. 有用户可见功能变化时同步更新 `CHANGELOG.md`。
5. 提交前检查 `git diff` 和 `git status`，不得覆盖无关的用户改动。

日志命名格式：`YYYY-MM-DD_NNN-short-topic.md`，同一天序号从 `001` 递增。

纯状态查询或未形成任何项目结论的简短问答可以不产生空日志。

## 开发规则

- 本地优先。项目文件、分析缓存和用户音频默认保存在本机。
- 使用中立的内部项目模型，不把核心数据结构绑定到某一家声库或编辑器。
- 第三方集成优先使用厂商公开的导入/导出格式、脚本 API 或插件 API；不要依赖脆弱的界面自动点击。
- 对自动分析的节拍、和弦、段落、音符等保存置信度和来源，并允许用户手工修正。
- 所有自动生成操作必须可撤销、可比较，并支持锁定局部结果后重生成。
- 不提交用户音频、商业声库、模型权重、密钥、许可证文件或第三方专有工程文件。
- 引入依赖或外部格式前，核对其官方文档、维护状态和许可证，并写入调研或决策记录。
- 未形成可运行工程前，不在文档中虚构安装、启动或测试命令。

## 完成标准

一轮实现只有在以下条件满足后才算完成：需求与示例已区分、变更已验证、日志已写、状态已更新、风险和未决事项已显式记录。
