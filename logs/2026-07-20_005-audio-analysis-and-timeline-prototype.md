## 子 Agent 分工与结论

- `audio_spike_impl`：独占实现 `tools/analyze_audio.py` 与 `analysis/README.md`。交付了只依赖 Python 标准库的 PCM WAV 分析基线；未读取真值、渲染清单或夹具 ID，实跑得到 119.993 BPM、0.970 秒首拍和 C major。
- `timeline_spec`：只读设计轨道、默认图层、选区、纠错和渐进呈现规则。主 Agent 采用了“测试夹具真值不得冒充算法准确率”“无人声伴奏不得显示为已识别演唱旋律”等意见，并写入原型和仓库说明。
- `audio_spike_review`：只读建立独立验收清单并复审最新版。确认速度、首拍、调性命中，段落主边界命中但存在额外候选，和弦 Top-1 未达到 0.9；同时指出输出覆盖、非原子写入、静音伪候选、非法参数和截断 WAV 风险。
- 审查 Agent 最初基于旧默认参数报告 60 BPM 和更低和弦结果；实现更新为 2048 FFT 和首拍对齐后，主 Agent 与审查 Agent 均重新实跑，因此旧数据未采用。

## 决定与理由

- 将当前 CLI 定位为透明技术基线，不定位为生产音乐信息检索算法。它已经证明本地 PCM 到可解释 JSON 的通路，但和弦与段落仍未通过夹具完整验收。
- 原型明确把夹具真值、算法候选和用户确认值分开，避免通过演示数据制造不存在的准确率印象。
- 首屏默认显示波形、能量、段落、和弦和节拍；频谱与精确算法参数渐进展开，保持新手可读与专业数据可访问。
- 无人声伴奏导入后，演唱草稿保持空状态；之后由用户选区和约束触发生成，不能把伴奏音符自动等同于人声主旋律。
