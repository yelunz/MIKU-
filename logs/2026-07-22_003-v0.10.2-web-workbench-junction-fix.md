# 轮 003 · v0.10.2 修复 web-workbench junction 失效导致打包旧文件

**日期**：2026-07-22
**阶段**：v0.10.1 后续修复 → v0.10.2
**目标**：用户报告 v0.10.0/v0.10.1 安装后"只看到 P1 本地优先技术原型，其他什么都没有"。排查发现根因是 `prototype/desktop-shell/web-workbench/` 是过时的物理副本而非 junction，导致 electron-builder 打包的是 2026/7/20 的旧 P5 版本 index.html。

## 用户问题

> 还是没有。这个版本和最开始让我用浏览器打开的版本看起来一模一样。然后会显示一个 P1 本地优先技术原型，其他什么都没有。

## 根因分析

### 根因：electron-builder 打包了过时的 web-workbench 物理副本

仓库中有两个 web-workbench 目录：

| 目录 | 类型 | 文件数 | index.html 大小 | app.js 大小 | 最后修改 |
|---|---|---|---|---|---|
| `prototype/web-workbench/` | 普通目录（源） | 11 | 25716 字节 | 295025 字节 | 2026/7/21 |
| `prototype/desktop-shell/web-workbench/` | **普通目录（旧副本）** | 5 | 15625 字节 | 208900 字节 | 2026/7/20 |

`.gitignore` 第 11 行写了 `web-workbench/` 应该是"由本地 junction/复制生成，不入库"，但实际上它是一个**独立的物理目录**，包含了 2026/7/20 的旧 P5 版本副本：

- 旧 index.html：没有 onboarding-panel、没有 P6-P9 容器、没有 v0.10.1 修复
- 旧 app.js：没有 onboarding 初始化、没有 P6-P9 模块挂载代码
- 缺少 6 个新 JS 文件：onboarding.js / stem-separator.js / transcription-panel.js / melody-generator.js / piano-roll-pro.js / error-recovery.js

electron-builder 的 `files: ["web-workbench/index.html", ...]` 实际打包的是 `prototype/desktop-shell/web-workbench/index.html`（旧版本），所以用户安装后看到的界面和"最开始浏览器版本一模一样"。

### 为什么 v0.10.0 和 v0.10.1 都有问题

- v0.10.0 打包时用的是旧 index.html（P5 版本，没有 P6-P9、没有 onboarding）
- v0.10.1 修复了 onboarding 和示例加载逻辑，但这些修复在 `prototype/web-workbench/` 里，没有同步到 `prototype/desktop-shell/web-workbench/`，所以打包的仍然是旧版本

### app.asar 内容验证

修复前用 `@electron/asar` 的 `listPackage` 检查 v0.10.2 第一次打包（junction 模式）的 app.asar：

```
total: 3
\main.js
\package.json
\preload.js
```

**只有 3 个文件！** web-workbench 的所有文件都没被打包。原因是 electron-builder 默认不跟随 junction/symlink。

## 修复方案

### 修复 1：删除 junction，改用普通目录 + 同步脚本

- 删除 `prototype/desktop-shell/web-workbench/` junction
- 用 `Copy-Item -Recurse` 把 `prototype/web-workbench/` 完整复制到 `prototype/desktop-shell/web-workbench/`
- 验证：`Get-Item` 显示 Mode 是 `d-----`（普通目录），LinkType 为空
- 验证：目录包含 11 个最新文件（app.js 295025 字节 / index.html 25716 字节等）

### 修复 2：新增同步脚本 `scripts/sync-web-workbench.bat`

为避免以后再次出现"修改了源目录但打包用的是旧副本"的问题，新增同步脚本：

```batch
@echo off
REM 同步 prototype/web-workbench/ 到 prototype/desktop-shell/web-workbench/
set "ROOT=%~dp0.."
set "SRC=%ROOT%\prototype\web-workbench"
set "DST=%ROOT%\prototype\desktop-shell\web-workbench"
if exist "%DST%" rmdir /s /q "%DST%"
xcopy "%SRC%\*" "%DST%\" /e /i /y /q
```

**用法**：每次修改 `prototype/web-workbench/` 后，打包前运行 `scripts\sync-web-workbench.bat` 同步到 desktop-shell 副本。

### 修复 3：版本号更新到 0.10.2

- `package.json`：version 0.10.1 → 0.10.2，description 更新
- `directories.output`：dist-v0.10.1 → dist-v0.10.2
- `.gitignore`：新增 `dist-v0.10.2/`
- 测试断言：版本 0.10.1 → 0.10.2，output dist-v0.10.1 → dist-v0.10.2

## 决定

1. **不用 junction**：electron-builder 25.1.8 默认不跟随 junction，导致 app.asar 缺少 web-workbench 文件。改用普通目录 + 同步脚本是最可靠的方案。`dereferenceSymlinks` 选项在 schema 校验阶段就被拒绝（不是合法选项）。
2. **同步脚本而非 npm hook**：不在 `npm run dist:win` 前自动同步，因为：
   - 同步是破坏性操作（删除并重建目录），应在用户知情下执行
   - 脚本 `scripts/sync-web-workbench.bat` 显式调用更清晰
   - 后续可考虑改为 `prebuild` npm script 自动化
3. **PyInstaller 不重新打包**：v0.10.2 只修复前端打包问题，分析服务后端无变化，复用 v0.10.0 的 PyInstaller 产物（288.13 MB）。
